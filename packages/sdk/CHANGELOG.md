## 0.3.0

### Minor Changes

- Move the Autopilot orchestration engine into the SDK, and split concrete
  notification channels into a new companion package.

  **`@driftwatch/sdk` additions** (all exported from the package root, zero new
  hard dependencies):

  - `MemoryStateStore` — the zero-dependency `StateStore` implementation.
  - `ApprovalService`, `AutopilotScheduler`, `executeControlAction` — the full
    perceive→reason→act orchestration engine (approval lifecycle, the scheduled
    drift-cycle loop, and control-action state transitions). These only ever
    called the `StateStore`/`Notifier` interfaces already in the SDK, so moving
    them out of the reference server required no new dependencies — they were
    pure orchestration all along.
  - `notifierForAction`, `safeNotify`, `notifyAll`, `NotifierRegistry` — dispatch
    helpers over the `Notifier` interface.
  - `RedisStateStore`, at the isolated subpath `@driftwatch/sdk/redis`. `ioredis`
    is an optional peer dependency — importing the SDK's package root never
    pulls it in; only code that imports `@driftwatch/sdk/redis` needs it
    installed.

  **New package `@driftwatch/autopilot`** — concrete `Notifier` implementations
  (`SlackNotifier`, `TelegramNotifier`, `WebhookNotifier`) and framework-agnostic
  inbound-webhook verification (`verifySlackSignature`, `parseSlackInteraction`,
  `verifyTelegramSecretToken`, `parseTelegramCallback`, `answerTelegramCallback`).
  Split out from the SDK core because these track Slack's and Telegram's APIs on
  their own schedule, independent of the SDK's release cadence — bring this in
  only if you use those channels.

  Anyone building their own server on top of the SDK now gets the full
  approval/scheduler/control-action engine for free, instead of having to
  reimplement it from the reference server's source as a template.

  `@driftwatch/server` is a consumer of both: it wires `ServerConfig` into
  concrete `RedisStateStore`/`MemoryStateStore` and
  `SlackNotifier`/`TelegramNotifier`/`WebhookNotifier` instances and hands them
  to the SDK's `ApprovalService`/`AutopilotScheduler`. No behavior change for the
  reference server.

## 0.2.1

### Patch Changes

- Fix behavioral drift detection against SigNoz and ensure custom metrics are actually recorded:

  - **Custom metrics are now recorded.** `agent.tokens`, `agent.tool.calls`, and
    `agent.tool.duration` were created at module load, which (via the package
    barrel) runs before `bootstrapTelemetry` starts the SDK and registers the
    global `MeterProvider`. The Metrics API returns a permanent no-op meter in
    that window, so every measurement was silently dropped — no `agent.*` metrics
    ever reached the backend, even though traces and logs did. The instruments
    are now created lazily on first use, binding them to the real provider.
  - **`detectBehavioralDrift` queries now conform to SigNoz's v4 query_range API.**
    Each builder query carries the required `queryName`, and the metrics queries
    use the v4 aggregation shape (`type`, `temporality: 'Delta'`,
    `timeAggregation` / `spaceAggregation`) instead of the rejected v3
    `aggregateOperator`. The p95 latency query targets the exploded
    `agent.tool.duration.bucket` series, since SigNoz does not expose the
    histogram under its base name. Previously every drift query failed with a 400
    or 500.

## 0.2.0

### Minor Changes

- SigNoz/OpenTelemetry correctness fixes and OpenTelemetry 2.x upgrade:

  - **Metrics now export with delta temporality** (SigNoz's recommendation)
    instead of OpenTelemetry's default cumulative. `detectBehavioralDrift` sums
    `agent.tool.calls` / `agent.tokens` and takes p95 of `agent.tool.duration`
    over time windows — summing cumulative running totals produced meaningless,
    ever-growing numbers; delta makes the windowed queries correct.
  - **Added `TelemetryConfig.otlpHeaders`** (env `OTEL_EXPORTER_OTLP_HEADERS`,
    standard `k=v,k2=v2` form), attached to every OTLP export. Enables **SigNoz
    Cloud** via `signoz-ingestion-key=<key>`; self-hosted needs none.
  - **`bootstrapTelemetry` now exports logs** over OTLP (`/v1/logs`) alongside
    traces and metrics, and enables pino log↔trace correlation (`trace_id` /
    `span_id` on Fastify logs), so you can jump from a span to its logs.
  - Upgraded the OpenTelemetry stack to 2.x (`resources` / `sdk-metrics` 2.9,
    `sdk-node` / exporters / `sdk-logs` 0.220, `auto-instrumentations-node`
    0.78) and switched from the deprecated `new Resource()` to
    `resourceFromAttributes()`.

  Also folds in prior unreleased changes shipping in this release: the package
  rename from `@agentpulse/sdk` to `@driftwatch/sdk` (config API
  `AgentPulseConfigSchema` / `loadAgentPulseConfigFromEnv` →
  `DriftWatchConfigSchema` / `loadDriftWatchConfigFromEnv`),
  `TelemetryConfig.capturePayloads` (env `OTEL_CAPTURE_PAYLOADS`), and
  `bootstrapTelemetry`'s `{ registerShutdownHandlers: false }` option.
