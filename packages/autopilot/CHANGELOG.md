## 0.2.0

### Minor Changes

- Move the Autopilot orchestration engine into the SDK, and split concrete
  notification channels into a new companion package.

  **`@driftwatch/sdk` additions** (all exported from the package root, zero new
  hard dependencies):

  - `MemoryStateStore` — the zero-dependency `StateStore` implementation.
  - `ApprovalService`, `AutopilotScheduler`, `executeControlAction` — the full
    perceive→reason→act orchestration engine (approval lifecycle, the scheduled
    drift-cycle loop, and control-action state transitions). These only ever
    called the `StateStore`/`Notifier` interfaces already in the SDK, so moving
    them out of the reference server required no new dependencies — they were
    pure orchestration all along.
  - `notifierForAction`, `safeNotify`, `notifyAll`, `NotifierRegistry` — dispatch
    helpers over the `Notifier` interface.
  - `RedisStateStore`, at the isolated subpath `@driftwatch/sdk/redis`. `ioredis`
    is an optional peer dependency — importing the SDK's package root never
    pulls it in; only code that imports `@driftwatch/sdk/redis` needs it
    installed.

  **New package `@driftwatch/autopilot`** — concrete `Notifier` implementations
  (`SlackNotifier`, `TelegramNotifier`, `WebhookNotifier`) and framework-agnostic
  inbound-webhook verification (`verifySlackSignature`, `parseSlackInteraction`,
  `verifyTelegramSecretToken`, `parseTelegramCallback`, `answerTelegramCallback`).
  Split out from the SDK core because these track Slack's and Telegram's APIs on
  their own schedule, independent of the SDK's release cadence — bring this in
  only if you use those channels.

  Anyone building their own server on top of the SDK now gets the full
  approval/scheduler/control-action engine for free, instead of having to
  reimplement it from the reference server's source as a template.

  `@driftwatch/server` is a consumer of both: it wires `ServerConfig` into
  concrete `RedisStateStore`/`MemoryStateStore` and
  `SlackNotifier`/`TelegramNotifier`/`WebhookNotifier` instances and hands them
  to the SDK's `ApprovalService`/`AutopilotScheduler`. No behavior change for the
  reference server.
